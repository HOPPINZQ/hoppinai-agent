# s02: Tool Use (工具使用)

`s01 > [ s02 ] s03 > s04 > s05 > s06 | s07 > s08 > s09 > s10 > s11 > s12`

> *"加一个工具, 只加一个 handler"* -- 循环不用动, 新工具注册进 dispatch map 就行。

## 问题

只有 `bash` 时, 所有操作都走 shell。`cat` 截断不可预测, `sed` 遇到特殊字符就崩, 每次 bash 调用都是不受约束的安全面。专用工具 (`read_file`, `write_file`) 可以在工具层面做路径沙箱。

关键洞察: 加工具不需要改循环。

## 解决方案

```
+--------+      +-------+      +----------------------+
|  User  | ---> |  LLM  | ---> | Tool Dispatch        |
| prompt |      |       |      | {                    |
+--------+      +---+---+      |   bash:       bash   |
                    ^           |   read_file:  read   |
                    |           |   write_file: write  |
                    +-----------+   edit_file:  edit   |
                    tool_result |   list_files: list   |
                                |   content_search:   |
                                |                   } |
                                +----------------------+

The dispatch map is a dict: {tool_name: handler_function}.
One lookup replaces any if/elif chain.
```

## 工作原理

### 1. 工具定义 -- 每个工具 = Schema + Handler

Agent02 引入了 `ToolDefinition` 类, 将工具的 Schema 描述和执行函数封装在一起。每个工具由五个要素组成:

```java
public class ToolDefinition {
    private String name;          // 工具名称
    private String description;   // 工具描述 (LLM 用来理解工具用途)
    private Tool.InputSchema inputSchema;  // JSON Schema 参数定义
    private Class<?> type;        // 参数类型 (用于反序列化)
    private Function<String, String> function;  // 处理函数
    private TypedToolInvoker typedInvoker;       // 类型化调用器
}
```

以 `read_file` 为例:

```java
public static ToolDefinition ReadFileDefinition = new ToolDefinition(
        "read_file",
        "读取指定相对文件路径的内容。适用于需要查看文件内容的场景，支持查看各种类型文件的完整内容。",
        createInputSchema(
                Map.of("path", createProperty("string", "工作目录中文件的相对路径。")),
                List.of("path")    // required 字段
        ),
        ReadFileInput.class,      // 参数类型
        Tools::readFile           // 方法引用作为 handler
);
```

**代码解释**:
- `createInputSchema()`: 辅助方法, 将 `Map<String, Object>` 转换为 Anthropic API 所需的 `Tool.InputSchema`
- `createProperty()`: 辅助方法, 创建 `{type, description}` 属性定义
- `ReadFileInput.class`: 告诉框架如何将 JSON 反序列化为 Java 对象
- `Tools::readFile`: 方法引用, 等价于 `input -> Tools.readFile(input)`

### 2. 注册 6 个工具

Agent02 在 `main()` 中注册了 6 个工具 -- 相比 Agent01 的 1 个, 扩展了文件操作能力:

```java
List<ToolDefinition> tools = new ArrayList<>();
tools.add(BashDefinition);          // bash - 执行 Shell 命令
tools.add(EditFileDefinition);      // edit_file - 编辑文本文件
tools.add(WriteFileDefinition);     // write_file - 写入文件
tools.add(ReadFileDefinition);      // read_file - 读取文件
tools.add(ListFilesDefinition);     // list_files - 列出文件和目录
tools.add(ContentSearchDefinition); // content_search - 搜索代码内容
```

每个工具的参数 Schema:

| 工具 | 必需参数 | 可选参数 | 说明 |
|------|---------|---------|------|
| `bash` | `command`, `type` | -- | `type` 可选 cmd/powershell/bash |
| `read_file` | `path` | -- | 读取文件完整内容 |
| `write_file` | `path`, `content` | -- | 写入文件, 不存在则创建 |
| `edit_file` | -- | `path`, `oldStr`, `newStr` | 替换文件中的文本片段 |
| `list_files` | -- | `path`, `fileType` | 列出目录内容, 支持按扩展名过滤 |
| `content_search` | `pattern` | `path`, `fileType`, `caseSensitive` | 基于 ripgrep 的正则搜索 |

**注意**: `edit_file` 和 `list_files` 没有必需参数 -- LLM 会根据上下文自行决定传哪些参数。

### 3. 三路分派 -- invoke()

Agent02 的核心创新是 `invoke()` 方法, 支持三种调用模式:

```java
public String invoke(Object convertedInput) throws Exception {
    // 路径 1: 类型化调用器 (最高优先级)
    if (typedInvoker != null) {
        return typedInvoker.apply(convertedInput);
    }
    // 路径 2: String 函数, 直接传入原始 JSON
    if (function != null) {
        if (convertedInput instanceof String) {
            return function.apply((String) convertedInput);
        }
        // 路径 3: 非 String 输入, 先序列化为 JSON 再调用
        return function.apply(OBJECT_MAPPER.writeValueAsString(convertedInput));
    }
    throw new IllegalStateException("没有该工具的处理方法: " + name);
}
```

**代码解释**:
- **路径 1** `typedInvoker`: 通过 `TypedToolFunction<T>` 接口创建, 支持泛型, 自动类型转换
- **路径 2** `function(String)`: 接收原始 JSON 字符串, handler 自行反序列化
- **路径 3** `function(Object→JSON)`: 输入被序列化为 JSON 后再传给 handler
- 三路分派保证了向后兼容 -- 新旧两种工具注册方式都能正常工作

### 4. 类型化接口 -- TypedToolFunction<T>

Agent02 引入了泛型函数式接口, 让工具 handler 可以直接接收反序列化后的 Java 对象:

```java
@FunctionalInterface
public interface TypedToolFunction<T> {
    String apply(T input) throws Exception;
}

@FunctionalInterface
public interface TypedToolInvoker {
    String apply(Object input) throws Exception;
}
```

构造器自动将 `TypedToolFunction<T>` 包装为 `TypedToolInvoker`:

```java
public <T> ToolDefinition(String name, String description,
        Tool.InputSchema inputSchema, Class<T> type,
        TypedToolFunction<T> typedFunction) {
    this.typedInvoker = input -> typedFunction.apply(type.cast(input));
}
```

这意味着 handler 方法可以直接声明参数类型, 无需手动反序列化:

```java
// 旧方式: 需要在 handler 内部反序列化
new ToolDefinition("read_file", ..., ReadFileInput.class,
    (String json) -> {
        ReadFileInput input = mapper.readValue(json, ReadFileInput.class);
        return doRead(input);
    })

// 新方式: 框架自动反序列化, 直接拿到类型安全的对象
new ToolDefinition("read_file", ..., ReadFileInput.class,
    (TypedToolFunction<ReadFileInput>) input -> doRead(input))
```

### 5. 组合模式 -- 不改循环

Agent02 和 Agent01 一样使用**组合模式**, 不继承 ZQAgent, 而是在 main 中创建实例:

```java
public class Agent02 {
    public static void main(String[] args) {
        AnthropicClient client = AnthropicOkHttpClient.builder()
                .apiKey(API_KEY).baseUrl(BASE_URL).build();

        List<ToolDefinition> tools = new ArrayList<>();
        tools.add(BashDefinition);
        tools.add(EditFileDefinition);
        tools.add(WriteFileDefinition);
        tools.add(ReadFileDefinition);
        tools.add(ListFilesDefinition);
        tools.add(ContentSearchDefinition);

        ZQAgent agent = new ZQAgent(client, MODEL, tools);
        agent.setSystemPrompt(buildSystemPrompt());
        agent.run();
    }
}
```

循环体与 s01 完全一致 -- 工具从 1 个变成 6 个, 但 `ZQAgent.run()` 一行代码都没改。

## 变更内容

| 组件 | 之前 (s01) | 之后 (s02) |
|-----|------|------|
| Tools | 1 (仅 bash) | 6 (bash, read_file, write_file, edit_file, list_files, content_search) |
| Tool Definition | 无独立类 | `ToolDefinition` 类 (name + schema + handler) |
| 类型安全 | 无 | `TypedToolFunction<T>` + `TypedToolInvoker` 接口 |
| 调用机制 | 硬编码 | `invoke()` 三路分派 |
| 路径安全 | 无 | 工作目录路径解析 + normalize |
| Agent loop | `while(true)` + `hasToolUse` | **不变** |
